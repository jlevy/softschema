# Publishing

softschema ships two packages that release **together, under the same version number**:

- **PyPI**: `softschema` (Python/Pydantic), built from `packages/python`.
- **npm**: `softschema` (TypeScript/Zod), built from `packages/typescript`.

A protected `vX.Y.Z` or `vX.Y.Z-rc.N` Git tag drives both.
The Python wheel and sdist derive their version from the tag via
`uv-dynamic-versioning`; the npm package carries the same version in
`packages/typescript/package.json`. Pushing the protected tag triggers
[`publish.yml`](../.github/workflows/publish.yml), which builds both packages once,
smokes those exact bytes, stages and attests them on a protected GitHub draft, and
publishes them via **trusted publishing (OIDC)**. The workflow makes that existing draft
public only after both registries and their provenance verify successfully.
No package-registry API tokens are stored in the repository.

## Versioning Convention

- **`release-metadata.json` owns the logical release.** Update its logical version,
  ecosystem artifact versions, state, conformance status, runtime bounds, and exact
  expected artifact names in the release PR. Update `packages/typescript/package.json`
  to the same npm SemVer.
  Keep source bootstrap `pin` values on the last verified stable release until both
  registries pass post-publish verification.
  The Python mapping is `X.Y.Z` for a stable release and `X.Y.ZrcN` for logical
  `X.Y.Z-rc.N`.
- **Executable agent bootstrap uses immutable pins.** Generate Python and npm bootstrap
  selectors from the release metadata.
  Every public executable example uses those exact pins; the repositoryâ€™s dependency
  cool-off does not govern another consumerâ€™s resolver and is not a reproducibility or
  security guarantee.
- **Registry package pages describe the artifact being published.** PyPI uses
  `packages/python/README.md` and npm uses `packages/typescript/README.md`; their
  install commands use the candidate `version`. Root docs and skills use only the
  verified `pin`, so a failed or partial release cannot poison bootstrap discovery.
- The committed skill mirrors under `.agents/` and `.claude/` are regenerated by
  `softschema skill --install`; a drift test keeps them in sync with the source.
- Patch bumps (`0.1.Z`) cover docs-only changes and small additive features.
  Reserve minor bumps (`0.Y.0`) for changes that meaningfully shift the API or spec.

## Trusted Publishing Requirements

The release workflow requires both registries to authenticate via OIDC from
`publish.yml`; it does not accept `*_TOKEN` secrets.
Before a release, a credentialed maintainer must verify these live settings:

**PyPI** (project-level trusted publisher):

- project name: `softschema`
- owner/repo: `jlevy/softschema`
- workflow: `publish.yml`
- environment: `pypi`

**npm** (package-level trusted publisher at npmjs.com â†’ `softschema` â†’ Settings â†’
Trusted Publishing):

- owner/repo: `jlevy/softschema`
- workflow: `publish.yml`
- environment: `npm`

The npm job requests an OIDC `id-token`, uses the npm bundled with pinned Node, and
publishes the already tested tarball with lifecycle scripts disabled.
Both auth and the registry provenance attestation are OIDC-based, so no token is
involved. The PyPI job likewise receives only the tested wheel and sdist.

## npm Trusted Publishing

Both packages are designed to publish from the release workflow over OIDC, with
provenance and no stored token.
The npm trusted publisher must exactly name `jlevy/softschema`, workflow `publish.yml`,
and environment `npm` before a protected tag can publish PyPI and npm in one run.

Repository files cannot prove that registry-side authorization is current.
Complete the credentialed publisher audit and protected-environment preflight before
publishing.
The one-time npm package bootstrap procedure, if it is still needed, lives in
[Publishing npm (bootstrap runbook)](publishing-npm.md); do not infer its completion
from this repository.

## Release Checklist

A release is **staged through a pull request**, not pushed straight to `main`: that way
CI runs the full matrix (lint, both test suites, the golden corpus on every runtime, and
the cross-impl parity diff) against the exact tree that will be tagged, before anything
is published. The tag is created on the merge commit only after that PR is green and
merged.

For each stable or release-candidate version:

1. **Branch from current `main`:** use a short-lived `codex/release-X.Y.Z` branch.

2. **Prepare immutable release metadata.** Set `release_state` to `candidate`, update
   the logical version and both ecosystem artifact versions, set conformance
   availability, and list every expected primary subject.
   Update `packages/typescript/package.json`, `CHANGELOG.md`, and generated bootstrap
   artifact-version claims in the same commit.
   Leave source quickstarts and skill bootstrap pins on the prior dual-registry-verified
   stable release. Do not hand-edit `build-metadata.json` by hand.
   After changing logical metadata, regenerate the committed development identity with:

   ```bash
   uv run python devtools/release.py build-metadata build-metadata.json \
     --source-commit 0000000000000000000000000000000000000000
   ```

   The protected-tag workflow replaces that development identity with the tagged source
   commit and conformance digest before either package build.

3. **Run the full validation pass and a dry release** from the
   [end-to-end testing runbook](e2e-testing.runbook.md): the local automated sweep
   (Phase 1), the clean-environment artifact smoke that CI also runs (Phase 2), and the
   manual quickstart and skill bootstrap checks CI cannot cover (Phases 3â€“4). Everything
   must exit 0. Dispatching `publish.yml` manually exercises the infrastructure path
   with development metadata but makes both registry jobs ineligible.
   It is not a release-candidate rehearsal: the protected-tag run is the first path that
   requires candidate state and exact ecosystem filenames.

4. **Open the release PR and merge once CI is green.** Push the branch, open a PR
   (`release: bump to X.Y.Z`), and wait for every required check to pass before merging
   to `main`. CI on the PR is the gate; do not tag a tree CI has not validated.

5. **Verify live release controls.** Confirm `main` and `v*` protection, full-SHA action
   enforcement, the `pypi` and `npm` environment restrictions, both matching
   trusted-publisher claims, the separate `github-release` approval environment, and
   repository-level immutable releases.
   See [Supply-chain security](../SUPPLY-CHAIN-SECURITY.md).
   Treat drift as a blocker.

   Check the immutable-release prerequisite without changing repository state:

   ```bash
   gh api -H 'X-GitHub-Api-Version: 2026-03-10' \
     repos/jlevy/softschema/immutable-releases --jq .enabled
   ```

   It must print `true`. Enabling the setting is an administrator action and must be
   completed before tagging.
   The protected workflow repeats this check before creating or modifying a GitHub
   release and fails closed when the setting is disabled.

   Re-read `github-release` through the repository environment API. It must already
   exist, disallow administrator bypass, require a reviewer, and restrict deployments to
   protected `v*` release tags before the workflow is allowed to rely on it:

   ```bash
   gh api -H 'X-GitHub-Api-Version: 2026-03-10' \
     repos/jlevy/softschema/environments/github-release
   gh api -H 'X-GitHub-Api-Version: 2026-03-10' \
     repos/jlevy/softschema/environments/github-release/deployment-branch-policies
   ```

   A workflow reference does not prove these protections are configured.
   Treat a missing or unprotected environment as a release blocker and preserve the
   authenticated responses with the release evidence.

6. **Tag the merge commit on `main` and push the tag.** The working tree must be clean
   and the tag must sit on the merged release commit, so the Python version derived from
   git is exactly `X.Y.Z` (the publish workflowâ€™s tag-vs-build guard fails otherwise):

   ```bash
   git checkout main && git pull origin main
   git tag -a vX.Y.Z -m "Release X.Y.Z"
   git push origin vX.Y.Z
   ```

   Never use `gh release create` to bypass tag protection.
   A tag that cannot be pushed through the protected path is not authorized to publish.

7. **Watch the protected release state machine.** Preflight and smoke freeze one
   candidate. The workflow then creates or reuses a GitHub draft, requires its exact
   asset inventory, attests the primary manifest subjects and a deterministic recovery
   bundle, classifies both registries, publishes only missing exact bytes, performs
   bounded digest/provenance verification, and finally publishes the existing draft.
   A conflicting same-version byte fails closed.
   `devtools/release_state.py` owns those classifications.

   The recovery bundle contains the complete original candidate transfer, including its
   recursive checksums, locked npm consumer, smoke support, and frozen standard-library
   state driver. If the 90-day Actions artifact expires after the draft is frozen, a
   rerun of failed jobs downloads the attested recovery assets, verifies their workflow,
   tag, and source-commit identity, restores the original bytes without rebuilding, and
   rechecks the transfer inventory before registry access.
   Rerun failed jobs rather than restarting preflight when recovering a partial release.

   GitHub cannot make bytes durable before the protected draft job is approved: the
   Actions artifact is the only copy available to that job before its first release
   mutation. Approve the initial `github-release` environment gate within 90 days.
   Once the exact draft assets exist, recovery no longer depends on Actions retention.

   ```bash
   RUN_ID=123456789
   gh run watch "$RUN_ID" --exit-status
   ```

   A manual `workflow_dispatch` is build-and-smoke-only.
   It never creates a release or publishes a registry subject, so it is not equivalent
   to the protected-tag path.

8. **Independently verify both registries** have the new version, then smoke-test the
   published artifacts (end-to-end runbook Phase 5):

   ```bash
   cd /tmp                                   # outside the repo: do not inherit its [tool.uv] cool-off
   ts="$(date -u +%Y-%m-%dT%H:%M:%SZ)"       # cutoff = now, so a same-day publish is included
   uvx --refresh --exclude-newer-package "softschema=$ts" \
     --from 'softschema==X.Y.Z' softschema --version
   npx --yes softschema@X.Y.Z --version
   ```

   The `--exclude-newer-package` override is needed because a freshly published version
   sits inside the 14-day supply-chain cool-off; anyone consuming it under that policy
   sees the same friction for ~14 days, which is intentional.
   Three details matter for the smoke test: use a full timestamp at the current moment
   (a date-only `$(date +%F)` is midnight UTC and excludes a version published later the
   same day), run from outside the repo so uv does not apply the projectâ€™s own cool-off,
   and pass `--refresh` so uv does not serve a cached index that predates the publish
   (see runbook Phase 5).

   Allow for publish propagation: PyPI updates its JSON API
   (`pypi.org/pypi/softschema/json`) before the **simple index** uv resolves against
   (`pypi.org/simple/softschema/`), so the first `uvx` run right after the workflow
   turns green can briefly fail with `no version of softschema==X.Y.Z` even though the
   release is live. `--refresh` clears uvâ€™s own cache but not PyPIâ€™s CDN edge.
   If that happens, confirm the workflowâ€™s digest and provenance gates passed, wait for
   the simple index to list the new `.whl`/`.tar.gz`, and retry.
   npm propagates quickly and rarely shows this gap.

9. **Require the published GitHub release to retain the frozen state.** It must be
   immutable and have the expected stable or prerelease channel, the exact
   manifest-selected and recovery assets, and the protected-workflow attestations.
   A stable tag must be the exact `/releases/latest` result; a release candidate must
   not displace the stable latest release.
   Do not recreate the release, replace an asset, or move the tag after verification.

10. **Record verified publication in a follow-up PR.** Only after the GitHub release,
    PyPI, and npm all pass exact digest and provenance verification, set `release_state`
    to `released`. For a stable release, advance both `pin` values to the verified
    release; for a release candidate, retain the previous verified stable pair.
    Regenerate public claims, skill fixtures/mirrors, and build metadata, then rerun the
    complete gates. A partial or failed release leaves source bootstrap on the previous
    usable pair.
