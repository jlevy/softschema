## Notes

Investigated for 0.2.1: the one-char fix (date -> RFC3339 timestamp) is correct and silences the per-invocation uv warning, BUT it activates the strif cutoff, which forces a re-lock. Two outcomes, neither clean for a patch:
- Re-lock with NO global cutoff: uv pulls latest non-strif deps (e.g. basedpyright 1.39.8 dated 2026-06-14, pytest 9.1.0) — violates the repo's 14-day cool-off.
- Re-lock under a global cutoff: versions are preserved but uv bakes [options].exclude-newer into uv.lock, which then mismatches plain 'uv run' (no UV_EXCLUDE_NEWER env) and re-resolves on every local run.
The repo intentionally uses a DYNAMIC CI cutoff (UV_EXCLUDE_NEWER=14-days-ago) and commits a lock with no [options] block, so no committed-lock state both respects cool-off and stays stable locally. Resolving this is a maintainer decision about lock cutoff-recording strategy (relates to ss-l592). Deferred from 0.2.1. CI is unaffected (uv steps are not --locked; only bun uses --frozen-lockfile).
